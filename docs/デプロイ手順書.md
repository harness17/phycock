## 2. 前提条件

### サーバー要件

| 要件 | 確認方法 |
|------|---------|
| .NET 10.0 ランタイムがインストール済み | `dotnet --version` |
| IIS がインストール・起動済み | IIS マネージャー確認 |
| IIS の `AspNetCoreModuleV2` がインストール済み | .NET Hosting Bundle インストール後に確認 |
| SQL Server が起動済み | SQL Server Management Studio で接続確認 |
| `Phycock` AppPool が作成済み | IIS マネージャー > アプリケーション プール |

### AppPool 設定

| 設定項目 | 値 |
|---------|-----|
| .NET CLR バージョン | マネージ コードなし |
| マネージ パイプライン モード | 統合 |
| ID | ApplicationPoolIdentity または専用サービスアカウント |

## 3. IIS デプロイ手順

### Step 1: Release ビルド

```bash
# Phycock/ ディレクトリで実行
cd Phycock
dotnet publish -c Release -o C:/deploy/Phycock
```

ビルド成果物は `C:/deploy/Phycock` に出力される。

**ビルド成果物の確認事項**:
- `Phycock.exe` が存在すること
- `web.config` が生成されていること
- `wwwroot/` フォルダが含まれていること

---

### Step 2: AppPool の停止

IIS で稼働中のアプリを安全に停止する。

```powershell
# PowerShell（管理者権限で実行）
Import-Module WebAdministration
Stop-WebAppPool -Name 'Phycock'
Write-Host "AppPool 停止完了"
```

AppPool の状態確認：

```powershell
Get-WebAppPoolState -Name 'Phycock'
# 出力: Stopped
```

---

### Step 3: ファイルの配置

既存ファイルを新しいビルド成果物で上書きする。

```powershell
# 既存ファイルをバックアップ（任意・ロールバック用）
$backupPath = "C:/deploy/backup/Phycock_$(Get-Date -Format 'yyyyMMdd_HHmmss')"
Copy-Item -Path "C:/inetpub/wwwroot/Phycock" -Destination $backupPath -Recurse
Write-Host "バックアップ完了: $backupPath"

# 新しいファイルをコピー
Copy-Item -Path "C:/deploy/Phycock/*" -Destination "C:/inetpub/wwwroot/Phycock" -Recurse -Force
Write-Host "ファイル配置完了"
```

**注意**: `Uploads/` フォルダ（ユーザーがアップロードしたファイル）は上書き対象から除外すること。

```powershell
# Uploads フォルダは除外してコピー（ユーザーデータ保護）
robocopy "C:/deploy/Phycock" "C:/inetpub/wwwroot/Phycock" /E /XD Uploads
```

---

### Step 4: 設定ファイルの確認

本番環境固有の設定を確認・更新する。

**`appsettings.json`** の確認項目：

```json
{
  "ConnectionStrings": {
    "PhycockConnection": "Server=<本番サーバー>;Database=PhycockDB;..."
  },
  "Smtp": {
    "Host": "smtp.example.com",
    "Port": 25,
    "EnableSsl": true
  },
  "Logging": {
    "LogLevel": {
      "Default": "Information"
    }
  }
}
```

> **注意**: `appsettings.json` には機密情報（接続文字列・パスワード）を含める場合があるため、本番サーバー上のファイルを直接編集するか、環境変数または `appsettings.Production.json` で上書きすること。

---

### Step 5: AppPool の起動

```powershell
Start-WebAppPool -Name 'Phycock'
Write-Host "AppPool 起動完了"

# 起動確認
Start-Sleep -Seconds 3
Get-WebAppPoolState -Name 'Phycock'
# 出力: Started
```

---

### Step 6: 動作確認

ブラウザまたは curl でアクセスを確認する。

```powershell
# HTTP レスポンス確認
$response = Invoke-WebRequest -Uri "http://localhost/Phycock" -UseBasicParsing
Write-Host "ステータスコード: $($response.StatusCode)"
# 期待値: 200 または 302（ログインページへリダイレクト）
```

## 4. DB 初期化・マイグレーション

### 新規環境セットアップ

初めてデプロイする環境（DB が存在しない場合）は EF Core Migrations を適用する。

```bash
cd H:/ClaudeCode/Phycock && dotnet ef database update --project Phycock
```

**実行内容**:
1. `PhycockDB` データベースの作成とマイグレーション適用
2. アプリ起動時に `Program.cs` の `MigrateAsync` が未適用マイグレーションを自動適用
3. アプリ起動時に `SeedAsync` が初期ロール・初期ユーザーを作成（ユーザーが 0 件のときのみ）

**初期ユーザー**:

| UserName | Password | Role |
|----------|----------|------|
| admin1@sample.jp | `Admin1!` | Admin |
| member1@sample.jp | `Member1!` | Member |

---

### テーブル・カラム変更時

本プロジェクトは **EF Core Migrations** でスキーマを管理する。

| 変更内容 | 手順 |
|---------|------|
| エンティティ追加・変更後 | `dotnet ef migrations add <名前> --project Phycock` でマイグレーション生成 |
| DB への適用 | `dotnet ef database update --project Phycock` で適用（または次回アプリ起動時に自動適用） |
| マイグレーション取り消し | `dotnet ef migrations remove --project Phycock` で最後のマイグレーションを削除 |

> **Note**: アプリ起動時に `MigrateAsync` が自動実行されるため、本番環境ではデプロイ後の初回起動でマイグレーションが自動適用される。

## 5. 動作確認

デプロイ完了後に以下を確認する。

### チェックリスト

| 確認項目 | 方法 |
|---------|------|
| トップページが表示される | ブラウザでアクセス |
| ログインできる | admin1@sample.jp / Admin1! でログイン |
| DB 接続が正常 | ホーム画面のダッシュボードにデータが表示される |
| ファイルアップロードが動作する | テストファイルをアップロード |
| メール送信が動作する | メール送信サンプルから送信テスト |
| 承認ワークフローが動作する | 申請作成・承認の一連フロー確認 |

## 6. ロールバック手順

問題が発生した場合、バックアップから復元する。

```powershell
# AppPool 停止
Stop-WebAppPool -Name 'Phycock'

# バックアップから復元
$backupPath = "C:/deploy/backup/Phycock_<バックアップ日時>"
Copy-Item -Path "$backupPath/*" -Destination "C:/inetpub/wwwroot/Phycock" -Recurse -Force
Write-Host "ロールバック完了"

# AppPool 起動
Start-WebAppPool -Name 'Phycock'
```

## 7. トラブルシューティング

### 502.5 / 500 エラー

| 原因 | 確認方法 | 対処 |
|------|---------|------|
| .NET ランタイム未インストール | `dotnet --list-runtimes` | .NET Hosting Bundle を再インストール |
| web.config の設定不正 | IIS のエラーログ | `aspNetCore` ハンドラーを確認 |
| 接続文字列エラー | Windows イベントログ | `appsettings.json` の DB 接続文字列を確認 |

### AppPool がすぐ停止する

| 原因 | 確認方法 | 対処 |
|------|---------|------|
| アプリ起動エラー | `C:/inetpub/wwwroot/Phycock/logs/stdout_*.log` | stdout ログを確認 |
| 依存ライブラリ不足 | イベントビューアー | Publish 時に `--self-contained` を使用 |

### stdout ログを有効化する方法

`web.config` の `stdoutLogEnabled` を `true` に変更する（デバッグ時のみ）：

```xml
<aspNetCore processPath=".\Phycock.exe"
            stdoutLogEnabled="true"
            stdoutLogFile=".\logs\stdout" />
```

> ログ出力後は必ず `false` に戻すこと（パフォーマンス影響あり）。
